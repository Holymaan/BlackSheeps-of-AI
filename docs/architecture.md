A few notes on the parts that aren't obvious from the boxes alone.
The Projects service is the heart of the extensibility story. A Project is just a tuple of destination + time window + participant submissions + optimization parameters. A school project sets the destination to OŠ Spinut, time window to 7:30–8:00 arrival, max walk distance to 300 m, max ride time to 30 min. An Ultra Split project sets the destination to Park Mladeži, time window to 18:00–20:00, allows longer walks (adults), and runs in reverse for the return trip. Same engine, different parameters. For the hackathon, hard-code one school project — but build the model and the API as if there will be many.
The optimization engine is two steps, both feeding off Valhalla. First, cluster submitted home points into candidate stops constrained by a safety-aware walking distance from Valhalla's pedestrian profile (not Euclidean — a 200 m straight line that crosses Vukovarska doesn't count). Second, solve a Vehicle Routing Problem with the school start time as a hard constraint, bus capacity as a constraint, and total bus-distance as the objective. Minimum bus count falls out of solving for total distance with a per-bus fixed cost. Google OR-Tools has a clean .NET binding and a school-bus example you can copy.
Why Valhalla, specifically. PostGIS will give you distances; Valhalla gives you time-aware, mode-aware routing on the OSM road network. The same Valhalla instance answers "how long does a kid walk from this home to this stop" (pedestrian profile) and "how long does the bus drive between these stops" (bus/truck profile). One server, two profiles, both critical.
On the 8-hour split, three people:

Person 1 — frontend: parent form with map pin (Leaflet + OSM tiles), admin map view showing submissions, stops, and final routes as colored polylines. Stub the API with mocks until person 2 is ready.
Person 2 — .NET backend + DB: project model, submissions CRUD, geocoding wrapper, Valhalla integration, route storage. Skip real auth for the demo — a hardcoded admin token is fine.
Person 3 — optimization: the clustering + OR-Tools VRP. Develop against a JSON file of fake submissions first; only wire to the DB once person 2 has it stable. This is the highest-risk piece — start it first.

Pre-seed 60–100 fake parent submissions clustered around the chosen school's catchment area so the demo looks real. The "wow" moment of the pitch is the admin clicking "generate routes" and watching the map fill with three or four clean colored loops instead of a hundred scattered dots.
